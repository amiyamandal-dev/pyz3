Key Design Principles

  1. Clear Separation
  2. User docs in docs/
  3. Developer docs in docs/development/

  4. Examples separate from tests

  5. Single Source of Truth

  6. Template in ziggy-pydust-template/
  7. No duplicate template systems

  8. Cookiecutter for all project generation

  9. Modular Organization

  10. CLI commands in separate files
  11. Zig types in pydust/src/types/

  12. Clear module boundaries

  13. Developer Experience

  14. Comprehensive examples
  15. Detailed documentation
  16. Debugger configurations included
  17. Watch mode for rapid iteration

File Naming Conventions

  • Python files: lowercase_with_underscores.py
  • Zig files: lowercase.zig
  • Documentation: UPPERCASE_FOR_MAJOR.md, lowercase_for_guides.md
  • Tests: test_*.py for pytest discovery
  • Scripts: *.sh for shell scripts

Adding New Features

When adding features:

  1. Implementation
  2. Add code to appropriate module in pydust/ or pydust/src/
  3. Create tests in test/

  4. Add example in example/

  5. Documentation

  6. User guide in docs/
  7. Implementation notes in docs/development/

  8. Update README.md if it affects quick start

  9. Template

  10. Update ziggy-pydust-template/ if affects new projects

  11. Test template generation

  12. Testing

  13. Add to run_all_tests.sh if needed
  14. Ensure pytest discovers new tests

Maintenance

  • Keep docs/ clean - User-facing only
  • Archive in development/ - Implementation details
  • Update README - Reflect current capabilities
  • Version updates - Keep dependencies current via Renovate
